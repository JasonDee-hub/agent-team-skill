# 精简工程准则（开发主场内化）

开发路径的主控快路径与专家默认按本准则做事：**能不写就不写，能用现成就用现成，最短能干活的 diff 优先**——但绝不偷掉校验、安全、可访问性，也绝不偷懒理解问题。

能力吸收自 [ponytail](https://github.com/DietrichGebert/ponytail) 的工程哲学，**已内化为本包规则**；不依赖、不安装、不引用外部 skill。

**域外路径（Atlas grilling）不适用本文件。**

## 按角色内化

| 能力 | 谁执行 |
|------|--------|
| 低风险单文件快路径 | **主控**（默认强度：标准；仅限主 skill 的全部准入条件成立） |
| 实现时爬「精简梯子」 | **Kane**（默认强度：标准）；Orin 最小验证改码；Atlas 写一次性脚本 |
| Diff 过度工程审查 | **Reed**（与正确性/安全审查一并做） |
| 整库膨胀审计（只报告不改码） | 用户要「能删啥/哪里过度」→ **Reed** 或 **Mira** |
| 收割故意简化标记成台账 | 用户问「以前标过哪些捷径」→ **Mira** / **Atlas** grep `lean:` / `ponytail:` 注释 |

强度（写入派遣包 `constraints`，默认 **标准**）：

| 强度 | 行为 |
|------|------|
| **轻** | 按需求做，但用一行点出更懒的替代方案，让用户选 |
| **标准** | 强制爬梯子；最短 diff、最短说明 |
| **极致** | YAGNI 优先；先删再加；交付最小方案同时质疑多余需求 |

用户说「别偷懒 / 完整方案」→ 不做极简压缩，按完整要求做。

## 精简梯子（先理解，再爬）

先读懂任务与调用链，再停在第一档够用的梯子上：

1. 根本不需要存在？→ 跳过并一句话说明（YAGNI）
2. 仓库里已有？→ 复用
3. 标准库？→ 用
4. 平台原生（如 `<input type="date">`、CSS、DB 约束）？→ 用
5. 已安装依赖？→ 用；不为几行代码新加依赖
6. 能一行？→ 一行
7. 否则：最小能工作的代码

## 硬规则

- 禁止未请求的抽象 / 脚手架 / 「以后再说」的配置层
- **修根因不糊症状**：共享路径上一处修好，胜过每个调用点打补丁
- 故意简化使用目标语言合法的注释语法，并保留字面量 `lean:`，例如 `// lean:`、`# lean:`、`<!-- lean: -->`（兼容识别历史 `ponytail:` 标记）
- 交付：代码优先；未请求的长篇辩护删掉。模式：`[做了啥] → skipped: [X], add when [Y]`
- **永不偷懒**：信任边界校验、防丢数据的错误处理、安全、无障碍、用户明确要的完整方案
- **永不偷懒理解问题**：先追溯真实流程再选梯子；小 diff 打在错误位置不算精简
- **稳定优先**：最小 diff 不能牺牲兼容性、迁移安全、回滚路径、数据完整性或可观测性；相关风险要处理或在 `risks` 中明确暴露

## 过度工程审查标签（Reed / 审计）

一行一条：`文件:L行: <标签> <问题>. <替换>.`

- `delete:` 死代码、无用灵活性、投机功能 → 替换：无
- `stdlib:` 手写了标准库已有的 → 点名标准库 API
- `native:` 依赖/代码做了平台已有能力 → 点名原生能力
- `yagni:` 单实现接口、无人改的配置、单调用方分层
- `shrink:` 同逻辑更短写法 → 给出短式

整库审计：按「砍幅从大到小」排序；**只出报告，不改码**；要改再派 Kane。

## 主控调度

- 快路径由主控按默认标准强度直接执行；一旦超出单文件或触及风险排除项，立即升级路径，不为保留快路径拆分任务
- 派 Kane / 改码 Orin / 脚本 Atlas：默认带 `constraints: 精简 标准`（或轻/极致）
- 派 Reed：正确性/安全 + 过度工程审查
- 「整库能删啥」→ 审计报告；「列捷径债务」→ 扫 `lean:`/`ponytail:` 注释
- 域外 grilling：**不套**本准则
